## Simple Summary

A standard interface for tokens.


## Abstract

The following standard allows for the implementation of a standard API for tokens within smart contracts.
This standard provides basic functionality to transfer tokens, as well as allow tokens to be approved so they can be spent by another on-chain third party.


## Motivation

A standard interface allows any tokens on Ethereum to be re-used by other applications: from wallets to decentralized exchanges.


## Specification

   - The first postcondition ensures that if the `_value` of tokens is successfully transferred, the sender's balance decreases by `_value` unless it's a self-transfer, in which case it remains unchanged.
   - The second postcondition clarifies that in a self-transfer, even upon success, the balance doesn't change.

   - The fourth postcondition handles failure cases. Success must be false if the sender's balance was insufficient.

   - Similar logic as `transfer` but applied to the `_from` address. This includes adjusting for the authorized allowance with the sender.
   - It decreases `_from`'s balance by `_value` and increases `_to`'s balance if successful, keeping the authorization in check.
   - Handles self-transfer conditions.
   - Ensures that transaction failure results from insufficient balance or allowance.

3. **approve Function**:
   - Straightforward postcondition: sets the allowance to `_value`.
   - Guarantees success since setting an allowance doesn't rely on existing balance validation.

4. **balanceOf and allowance Functions**:
   - These view functions have postconditions that directly relate the return values to their respective mappings, ensuring correct state reading for queried balances or allowances.
