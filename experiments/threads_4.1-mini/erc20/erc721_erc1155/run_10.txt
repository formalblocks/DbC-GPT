<eip>
## Simple Summary

A standard interface for tokens.


## Abstract

The following standard allows for the implementation of a standard API for tokens within smart contracts.
This standard provides basic functionality to transfer tokens, as well as allow tokens to be approved so they can be spent by another on-chain third party.


## Motivation

A standard interface allows any tokens on Ethereum to be re-used by other applications: from wallets to decentralized exchanges.


## Specification

 - The following specifications use syntax from Solidity `0.4.17` (or above)
 - Callers MUST handle `false` from `returns (bool success)`.  Callers MUST NOT assume that `false` is never returned!

OPTIONAL - This method can be used to improve usability,

``` js
function decimals() public view returns (uint8)
```

#### transferFrom

Transfers `_value` amount of tokens from address `_from` to address `_to`, and MUST fire the `Transfer` event.

The `transferFrom` method is used for a withdraw workflow, allowing contracts to transfer tokens on your behalf.
This can be used for example to allow a contract to transfer tokens on your behalf and/or to charge fees in sub-currencies.
The function SHOULD `throw` unless the `_from` account has deliberately authorized the sender of the message via some mechanism.

*Note* Transfers of 0 values MUST be treated as normal transfers and fire the `Transfer` event.

``` js
function transferFrom(address _from, address _to, uint256 _value) public returns (bool success)
```

THOUGH The contract itself shouldn't enforce it, to allow backwards compatibility with contracts deployed before

#### allowance

``` js

```



### Events

The previous postcondition incorrectly assumed that the allowance (_allowed[_from][msg.sender]) is always decreased by _value on a successful transferFrom call, even in the case when _from == _to (self-transfer). However, in the ERC20 specification, a self-transfer should not change balances or allowances.

Therefore, the postcondition now explicitly handles the self-transfer case (_from == _to) by requiring that balances and allowance remain unchanged. For other cases (_from != _to), balances and allowance change accordingly.
